# 中文说明

## 这个 Skill 是什么

`top-tier-research-reviewer` 是一个面向科研工作的可复用 Skill，用来帮助 AI 助手以顶刊审稿人、顶会审稿人、博士生导师、建模专家和实验设计专家的视角，审查和改进论文、研究方案、实验设计、代码实现、rebuttal 和 PPT 内容。

它不是普通的“论文总结提示词”。它的目标是判断一项研究是否具备清晰的科学问题、真实成立的创新点、严谨的方法设计、可信的实验验证、可复现的代码实现，以及足够支撑顶刊/顶会投稿的表达质量。

## 为什么要做这个 Skill

很多科研辅助 prompt 容易停留在表层：总结论文、罗列优点、泛泛建议“增加实验”。这个 Skill 更强调审稿式判断：

- 区分作者声称的创新和实际成立的创新；
- 区分科学贡献、工程组合、实现细节和可迁移思想；
- 检查每个实验是否真正验证了对应创新点；
- 发现 baseline 不强、消融实验不完整、数据泄漏、未来信息泄漏、结果过度解读等问题；
- 明确判断论文当前更接近普通期刊、SCI 二区、SCI 一区、顶刊候选、workshop、普通会议还是顶会候选；
- 给出可以直接执行的修改方案，而不是泛泛而谈。

## 适用场景

这个 Skill 可以用于：

- 论文精读；
- 顶刊/顶会审稿式评价；
- SCI 论文投稿前自查；
- 研究创新点设计；
- 实验设计和 baseline 检查；
- 消融实验、鲁棒性实验、极端场景实验设计；
- 代码和论文一致性检查；
- 时序预测数据泄漏和未来信息检查；
- rebuttal 和 revision 策略；
- 将复杂算法或论文框架压缩成一页 PPT。

## 重点支持的研究方向

这是一个通用的顶刊/顶会科研审稿 Skill，不局限于某个方向。它尤其适合：

- AI / 机器学习；
- 时间序列预测；
- 时空图神经网络；
- 优化、调度、控制和决策系统；
- 仿真驱动研究；
- 交通—能源融合；
- 电动车充电需求预测；
- 移动补能和储能优化；
- SUMO 交通仿真；
- 配电网、电压支撑、P/Q 服务；
- 交叉工程系统研究。

## 核心工作模式

### 1. 论文精读模式

输入一篇论文后，Skill 会固定分析：

- 论文解决什么问题；
- 现有方法为什么不够；
- 作者声称的创新点；
- 实际成立的创新点；
- 哪些只是工程组合；
- 哪些宏观思想可以迁移；
- 哪些技术细节不适合直接照搬；
- 技术路线、变量、公式、算法和实验逻辑；
- 每个实验验证哪个创新主张；
- 对当前研究有什么借鉴意义。

### 2. 稿件审查模式

用于检查自己的论文、proposal、方法章节、实验章节或 discussion。它会输出：

- 总体判断；
- 投稿成熟度等级；
- 致命问题；
- 主要问题；
- 次要问题；
- 可直接执行的修改清单；
- 核心贡献的重构建议；
- 投稿前必须补充的实验；
- 如果要冲顶刊/顶会还差什么。

### 3. 创新设计模式

当你只有数据、场景或初步想法时，这个模式不会先推荐某个神经网络，而是先推导：

- 真实系统中的核心矛盾；
- 哪些变量不可观测、难预测或难控制；
- 现有研究默认了哪些不合理假设；
- 误差如何传导到下游决策；
- 是否存在交通、电力、服务质量、经济性等系统耦合；
- 应该做预测、反演、优化、控制、因果分析、韧性分析还是混合范式；
- 最小可发表版本、强期刊/会议版本、顶刊/顶会版本分别是什么。

### 4. 实验设计模式

强制建立“创新点—实验—指标”的对应关系。典型输出是：

| 创新主张 | 必须设计的实验 | 核心指标 | 必要 baseline | 预期证据 | 回应的审稿质疑 |
|---|---|---|---|---|---|

同时检查是否缺少：

- naive baseline；
- 统计模型；
- 传统机器学习模型；
- 深度学习 baseline；
- 当前强模型或代表性 SOTA；
- 消融实验；
- 参数敏感性；
- 多随机种子；
- 统计显著性；
- 鲁棒性；
- 极端场景；
- 真实约束；
- 数据泄漏和未来信息检查。

### 5. 代码—论文一致性模式

当输入代码和论文后，Skill 会检查：

- 论文中的变量是否在代码中实现；
- 公式是否对应 loss、objective、constraint 或 update rule；
- 代码是否用了论文没有说明的特征；
- 是否使用未来信息；
- train/validation/test 是否严格；
- scaler 是否只在训练集拟合；
- online adaptation 是否使用未来标签；
- 模型输入输出维度是否一致；
- 每个声称的模块是否真的参与 forward / inference / optimization；
- 消融实验是否真的关闭对应模块；
- 结果文件是否和论文表格一致。

### 6. Rebuttal 和 revision 模式

输入审稿意见后，Skill 会：

- 判断拒稿/大修/小修的核心原因；
- 聚类 reviewer concern；
- 区分致命问题和可修问题；
- 给出回复策略；
- 判断哪些问题靠文字修改即可，哪些必须补实验；
- 起草 point-by-point rebuttal；
- 评估修改后的风险。

### 7. PPT 压缩模式

当需要把论文方法或算法放进一页 PPT 时，Skill 会输出：

- slide title；
- 一句话目标；
- 3–5 个核心模块；
- 6–10 步压缩流程；
- 值得保留的公式；
- 应该删除的细节；
- 图示布局建议；
- 60–90 秒讲稿。

## 设计原则

这个 Skill 的核心原则是：

- 不只总结，要评价；
- 不只指出问题，要给修改方案；
- 不只看模型，要看科学问题；
- 不只看指标提升，要看实验是否支撑创新；
- 不只看论文写法，也要看代码和数据是否可信；
- 不把复杂模型等同于高水平贡献；
- 不把“有潜力”当作结论，必须给出明确等级判断。
